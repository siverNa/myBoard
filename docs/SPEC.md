# myBoard 프로젝트 명세서 v2

## 1. 프로젝트 개요

### 프로젝트명
**myBoard**

### 프로젝트 목적
Laravel 기반 게시판 서비스를 구현하여, 단순 CRUD를 넘어  
**채널 구조, 권한 분리, 채널 신청 및 승인 프로세스, 관리자 기능**을 포함한  
실무형 포트폴리오 프로젝트를 제작한다.

### 개발 목표
- Laravel 기반 웹 서비스 구현
- 게시판, 댓글, 채널, 관리자 기능 구현
- 권한 관리 구조 설계
- 유지보수가 쉬운 DB 구조 설계
- 포트폴리오 제출용 완성형 프로젝트 제작

---

## 2. 프로젝트 핵심 개념

이 프로젝트는 단순 게시판이 아니라 **채널 기반 게시판 서비스**다.

사용자는 메인 또는 채널 목록 화면에서 채널을 선택해 진입한다.  
채널에 진입한 뒤에는 그 채널 안에서 게시글 목록을 보고, 게시글을 작성한다.  
따라서 **게시글 작성 시 별도로 채널을 다시 선택하지 않는다.**

즉, 게시글은 항상 **현재 접속 중인 채널에 종속**된다.

## 3. 핵심 기능

### 기본 기능
- 회원가입
- 로그인 / 로그아웃

- 관리자 페이지 제공
- 메인 페이지 집계형 정보 제공

## 4. 사용자 권한 정의

권한은 **전역 권한**과 **채널 단위 권한**으로 구분한다.

### 4-1. 전역 권한

#### 일반 사용자 (`user`)
- 회원가입 후 기본적으로 부여되는 권한
- 채널 접속 가능
- 게시글 작성 가능
- 댓글 작성 가능
- 본인 게시글 수정/삭제 가능
- 본인 댓글 수정/삭제 가능
- 채널 개설 신청 가능

#### 최고 관리자 (`super_admin`)
- 전체 채널 관리 가능
- 전체 게시글 / 댓글 관리 가능
- 채널 개설 신청 승인 / 반려 가능
- 최고 관리자 권한 부여 가능
- 관리자 페이지 전체 접근 가능

---

### 4-2. 채널 단위 권한

#### 채널 관리자 (`channel_manager`)
특정 채널에 대해 부여되는 권한이다.

- 자신이 관리하는 채널 내 게시글 관리 가능
- 해당 채널의 게시글 삭제 또는 숨김 처리 가능
- 해당 채널 설정 일부 수정 가능
- 여러 채널의 관리자가 될 수 있음

즉, 사용자는 전역적으로는 일반 사용자이지만  
특정 채널에서는 채널 관리자일 수 있다.

## 5. 서비스 정책

### 5-1. 로그인 필요 기능
아래 기능은 로그인한 사용자만 가능하다.

- 게시글 작성
- 게시글 수정
- 게시글 삭제
- 댓글 작성
- 댓글 수정
- 댓글 삭제
- 채널 개설 신청

비로그인 사용자가 해당 기능을 누르면:
- 로그인 필요 안내 메시지를 표시하고
- 로그인 페이지로 이동시킨다

---

### 5-2. 게시글 작성 정책
게시글은 **채널 내부에서 작성**한다.

예를 들어 사용자가 A채널에 접속해 있다면,  
A채널의 게시글 작성 화면에서 등록한 글은 자동으로 A채널 소속 글이 된다.

따라서 게시글 작성 시:
- 별도의 채널 선택은 하지 않음
- 현재 접속한 채널 기준으로 저장
- 필요하면 해당 채널 내 카테고리만 선택 가능

---

### 5-3. 게시글 수정/삭제 정책
게시글은 아래 사용자만 수정/삭제할 수 있다.

- 게시글 작성자
- 최고 관리자
- 해당 채널 관리자

---

### 5-4. 댓글 수정/삭제 정책
댓글은 아래 사용자만 수정/삭제할 수 있다.

- 댓글 작성자
- 최고 관리자
- 해당 댓글이 속한 게시글의 채널 관리자

---

### 5-5. 채널 생성 정책
채널은 일반 사용자가 직접 생성하지 않는다.

- 사용자 또는 채널 관리자가 채널 개설 신청 가능
- 최고 관리자가 신청 내용을 검토
- 승인 시 채널 생성
- 신청자는 승인된 채널의 채널 관리자가 됨
- 기존 채널 관리자도 새로운 채널을 추가로 소유 가능

---

- 게시글 수가 많은 상위 채널 6개 표시
- 각 채널의 최신 게시글 일부 표시
- 로그인 여부에 따라 메뉴 다르게 표시

목적:
- 서비스 첫 진입점
- 활성 채널 및 최신 활동 확인

---

#### 2) 회원가입 페이지

- 아이디
- 이메일
- 비밀번호
- 비밀번호 확인

기능:
- 회원가입 처리
- 아이디 중복 검사
- 이메일 중복 검사

#### 3) 로그인 페이지
입력 항목:
- 아이디

- 이메일

정책:

#### 5) 비밀번호 재설정 페이지
입력 항목:
- 아이디

권장 방식:
- 확인 후 비밀번호 재설정 절차로 이동

---

### 6-2. 채널 관련 화면

#### 6) 채널 목록 페이지
기능:
- 전체 채널 목록 표시
- 채널명, 설명, 게시글 수 등 표시
- 채널 선택 시 해당 채널 상세로 이동

기능:
- 채널 기본 정보 표시

이 페이지에서 사용자가 글쓰기를 누르면  
**현재 채널 기준으로 게시글 작성 화면으로 이동**한다.

---

#### 8) 채널 개설 신청 페이지
입력 항목:
- 채널명
- 채널 설명

기능:

- 신청 상태는 기본 `pending`(대기)

---

#### 11) 게시글 작성 페이지
기능:
- 제목 입력
- 내용 입력
- 카테고리 선택

중요 정책:

#### 12) 게시글 수정 페이지

권한:

- 해당 채널 관리자

## 7. 기능 명세

### 7-1. 인증 기능

#### 회원가입
- 아이디, 이메일, 비밀번호 입력
- 아이디 중복 불가
- 이메일 중복 불가
- 비밀번호 최소 길이 적용

#### 로그인
- 아이디 + 비밀번호 방식
- 성공 시 메인 또는 이전 페이지로 이동
- 실패 시 오류 메시지 출력

#### 로그아웃
- 세션 종료
- 메인 페이지 이동

#### 아이디 찾기
- 이메일로 본인 확인
- 아이디 일부 표시 또는 안내 처리

#### 비밀번호 재설정
- 아이디와 이메일 확인
- 새 비밀번호 설정 절차 진행
- 가능하면 토큰 기반으로 구현

---

### 7-2. 채널 기능

#### 채널 조회
- 사용자는 채널 목록 조회 가능
- 특정 채널 선택 후 상세 페이지 접속 가능

#### 채널 개설 신청
입력값:
- 채널명
- 채널 설명
- 개설 사유

저장 정책:
- 신청 상태는 `pending`(대기)

#### 채널 승인
- 최고 관리자가 신청 승인
- 실제 채널 생성
- 신청자를 해당 채널의 관리자 권한으로 등록
- 신청 상태를 `approved`(승인)로 변경

#### 채널 반려
- 상태를 `rejected`(반려)로 변경
- 반려 사유 저장 가능

---

### 7-3. 게시글 기능

#### 게시글 등록
필수값:
- 제목
- 내용
- 카테고리

저장 정책:
- 현재 접속한 채널의 `channel_pk`를 자동 저장
- 사용자가 별도로 채널을 선택하지 않음

#### 게시글 수정
- 권한 확인 후 수정 가능

#### 게시글 삭제
- soft delete 적용 권장

#### 게시글 검색
검색 조건 예시:
- 제목
- 제목 + 내용
- 작성자

#### 게시글 목록 페이지네이션
- 기본 10개 또는 15개 단위 적용

---

### 7-4. 댓글 기능

#### 댓글 등록
- 로그인 필요
- 내용 필수

#### 댓글 수정/삭제
- 본인 또는 관리자 권한만 가능

## 8. 데이터베이스 설계 원칙

### 기본 원칙 1
모든 테이블의 기본키 컬럼명은 `id`가 아니라 **`pk`** 로 통일한다.

예:
- `users.pk`
- `channels.pk`
- `posts.pk`

### 기본 원칙 2
참조 컬럼은 어느 테이블을 참조하는지 명확히 표현한다.

예:
- `user_pk`
- `channel_pk`
- `post_pk`
- `comment_pk`
- `category_pk`

이렇게 하면 `id`라는 이름이 중복되어 헷갈리는 문제를 줄일 수 있다.

---

## 9. 데이터베이스 테이블 설계

### 9-1. users
사용자 기본 정보 테이블

주요 컬럼:
- `pk`
- `login_id`
- `email`
- `password`
- `global_role`
- `created_at`
- `updated_at`

설명:
- `global_role`은 `user`, `super_admin` 정도로 단순 운영

---

### 9-2. channels
채널 정보 테이블

주요 컬럼:
- `pk`
- `name`
- `description`
- `status`
- `created_user_pk`
- `created_at`
- `updated_at`

설명:
- `status`는 `active`, `inactive`
- 채널 생성자는 사용자 참조 컬럼으로 저장

---

### 9-3. categories
채널별 카테고리 테이블

주요 컬럼:
- `pk`
- `channel_pk`
- `name`
- `sort_order`
- `created_at`
- `updated_at`

설명:
- 하나의 채널은 여러 카테고리를 가질 수 있음
- 하나의 카테고리는 한 채널에 속함

---

### 9-4. posts
게시글 정보 테이블

주요 컬럼:
- `pk`
- `channel_pk`
- `category_pk`
- `user_pk`
- `title`
- `content`
- `view_count`
- `created_at`
- `updated_at`
- `deleted_at`

설명:
- 게시글은 항상 특정 채널에 속함
- 게시글 작성 시 채널은 현재 접속한 채널 기준으로 자동 지정
- soft delete 적용

---

### 9-5. comments
댓글 정보 테이블

주요 컬럼:
- `pk`
- `post_pk`
- `user_pk`
- `content`
- `created_at`
- `updated_at`
- `deleted_at`

---

### 9-6. channel_requests
채널 개설 신청 테이블

주요 컬럼:
- `pk`
- `applicant_user_pk`
- `channel_name`
- `channel_description`
- `reason`
- `status`
- `reviewed_user_pk`
- `reviewed_at`
- `reject_reason`
- `created_at`
- `updated_at`

상태값:
- `pending`
- `approved`
- `rejected`

## 14. 확정된 핵심 설계 요약

현재 기준으로 확정된 방향은 다음과 같다.

- 프로젝트명은 `myBoard`
- PHP 7 + Laravel + MySQL 사용
- 기본키 컬럼명은 모두 `pk` 사용
- 참조키 컬럼은 `user_pk`, `channel_pk` 형태로 명확히 작성
- 게시글은 채널 내부에서 작성되며, 작성 시 별도 채널 선택 없음
- 채널 안에서 카테고리를 선택해 게시글 작성 가능
- 권한은 최고 관리자 / 일반 사용자 / 채널 관리자로 구분
- 채널 생성은 신청-승인 방식으로 처리
- DB는 과도하게 복잡하지 않게 유지
